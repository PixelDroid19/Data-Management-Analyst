# Developer Docs Convention for DM Flows

## Purpose

This file defines **how to persist technical documentation for developers** when an SDD DM investigation must be stored inside the project.

This documentation **is not OpenSpec**.

- **OpenSpec** guides the AI while investigating.
- **Developer Docs** explain the flow to other people on the team.

Think of this as a **repo-local technical wiki** layer, inspired by approaches like DeepWiki, but stored directly in `docs/`.

The `docs/` folder does not need to exist beforehand. Create it inside the analyzed project only when an investigation must be persisted.

## Documentation output language

All skills, agents, shared guides, and templates must remain in English.

However, the persisted developer documentation generated by the SDD workflow must:

- default to **Spanish** when the user does not request a language
- use the **language explicitly requested by the user** when one is provided

This rule applies to the main Markdown document, summaries, section headings, notes, and embedded explanatory text.

## When to use this convention

Activate this convention by default for SDD investigations involving flows, DMs, payloads, channels, and views, because this repository persists that knowledge so other developers can reuse it later.

Also activate it when the user explicitly asks for persisted developer-facing output, for example:

- "document this flow"
- "leave it in docs"
- "generate a technical wiki"
- "I want a markdown file for other developers"
- "save it as project documentation"

Also activate it when the user makes a **comprehensive investigation request** that already behaves like a reusable piece of technical documentation — for example, when one request asks for the DM flow, the payload, what it receives, where it is used, how it is used, and where the view or route comes from.

A prompt that combines multiple DM concerns (payload, channels, usage, origin) must be treated as a `docs` signal by default.

Leave the result inline only when the user:

- explicitly asks for it
- is asking a small or highly specific question
- or is not requesting a reusable end-to-end investigation

If the request already looks like a complete developer-facing investigation, persist the output in `docs/` even if the user never literally writes the word "documentation".

In this repo, a request like "find the flow", "trace the DM", "tell me its payload", or "where does this view come from" should normally close with persisted documentation unless the user explicitly says they do not want files.

## Output folder

The documentation must be stored inside the project in:

- `docs/flows/` -> documentos principales
- `docs/flows/diagrams/` -> main `.mmd` files and, when applicable, `.png` or `.svg`

If that structure does not yet exist, create it as part of the persistence work. Do not depend on the repository already containing a pre-seeded `docs/` folder.

The HTML viewer is **pre-built** inside `skills/sdd-docs-viewer/assets/`. To generate it, run:

```bash
python skills/sdd-docs-viewer/scripts/deploy_viewer.py --target-repo <path>
```

**DO NOT generate HTML, JS, or CSS from scratch.** The viewer is always generated by running the script above. The AI must never waste tokens creating frontend code for the viewer.

When the investigation is persisted for an analyzed project, the default behavior must be to export the viewer into `docs/site/` inside that project, unless the user explicitly asks not to generate the web output.

Do not mark the documentation task as complete until the exported viewer artifacts are present. At minimum verify:

- `docs/site/index.html`
- `docs/site/app.js`
- `docs/site/styles.css`
- `docs/site/manifest.json`

## File format

Default format:

- `.md`

Optional format:

- `.mdx` when the project already uses MDX or the user explicitly asks for it

If there is no strong reason to use MDX, use **standard Markdown** to maximize compatibility.

## Naming convention

### Main document

```text
docs/flows/<slug>.md
```

Examples:

- `docs/flows/dm-simulacion-housing.md`
- `docs/flows/flow-home-loan-form.md`
- `docs/flows/dm-logdown-participants.md`

### Source diagrams

```text
docs/flows/diagrams/<slug>_<num>_<type>_<title>.mmd
```

### Rendered diagrams

```text
docs/flows/diagrams/<slug>_<num>_<type>_<title>.png
```

## What each document must contain

Every persisted developer document must include at least:

1. Executive summary (DM, topology, main logic)
2. Origin of the view (entry point, host page)
3. Component classification (origin: `node_modules` / app-local / platform)
4. DM technical profile (properties, methods, dependencies)
5. **Service call chain** (order of backend calls, responses, parameter mapping between services)
6. Logical branches and state machine
7. Payload (request and response shapes)
8. Channels (consumed and published)
9. Downstream navigation
10. DM reuse across the project
11. Error handling (codes, handlers, retry/logDown)
12. Critical findings (confirmed issues with severity)
13. Conclusion and open gaps

UI and visual elements should only be mentioned as entry-point context, not as the focus.

**Stay focused on the flow the user asked about.** Do not drift into tangential flows, unrelated DMs, or speculative connections.

## Evidence-first authoring pattern

Each persisted document should read like a compact technical wiki page, not like a raw terminal transcript.

Mandatory authoring pattern:

1. Start with a `<details>` block that lists the key source files (limit to 3-6 that matter most).
2. Immediately follow with the H1 title.
3. Add a short introduction before the deeper technical sections.
4. Use focused H2/H3 numbered sections.
5. Prefer tables and compact bullet lists over long paragraphs.
6. **Do NOT add source citations after every single section.** Only cite sources inline when the evidence is critical or non-obvious. The document should read cleanly, not be cluttered with references.

## Evidence legend

Persisted docs should make uncertainty visible without becoming noisy.

Use these evidence labels where they clarify important findings:

- `[CONFIRMED]` — directly read from source
- `[INFERRED]` — supported by confirmed evidence, but not read end-to-end as a direct token
- `[NOT FOUND]` — searched for explicitly and not found

Recommended usage:

- a small legend near the top of the document
- status columns in key tables
- inline labels for disputed or partial findings

Do not label every trivial sentence. Preserve the labels where they matter most: routes, host pages, DM methods, payload fields, channels, service-call steps, downstream destinations, and important branch outcomes.

Do not dump entire investigation notes verbatim into the final document.

The final doc should help a developer answer the question quickly, then drill deeper section by section.

## Recommended diagrams

For DM flow documentation, prioritize these diagrams:

### 1. Technical DM class diagram (MANDATORY)

Use it when the investigation opened the DM source code or the package source that implements the DM.

Purpose:

- show the DM as a technical artifact, not just as a flow step
- expose the properties a developer can set
- expose the public methods a developer can call
- expose emitted events / response events when visible
- expose dependencies on BGDM, BGADP, provider, helper builder, or service layer

Preferred format:

- `classDiagram` for the DM internals

Do not replace the technical DM diagram with prose alone when the source was inspected.

### 2. Sequence diagram (MANDATORY)

Use it to show the logical interaction chain between:

- orchestrator/host
- channels
- DM
- validators/helpers
- external services or BGADPs

For sequence diagrams:

- declare all participants at the top
- use Mermaid sequence arrows only
- use notes/alt/opt blocks when they clarify the branch behavior

### 3. Activity diagram (optional)

Use only when the DM flow has multiple significant decision branches worth visualizing.

Keep activity diagrams focused:

- around 10-12 nodes max
- short labels
- no HTML tags or improvised Mermaid syntax

## Fidelity rule

The documentation must come from validated evidence, not inference.

- Do not invent payloads.
- Do not invent channels.
- Do not promote an auxiliary DM to the principal DM.
- If evidence is missing, leave an explicit gaps section.
- Do not silently rewrite `[INFERRED]` findings as `[CONFIRMED]` during final prose cleanup.
- Do not place `[NOT FOUND]` items into diagrams as if they were real nodes or edges.

## Practical diagram rule

If you can validate and render the diagram, save:

- `.mmd`
- `.png` or `.svg`

If you cannot render it at that time, save at least:

- the Mermaid block inside the `.md`
- the source `.mmd` inside `docs/flows/diagrams/`

In both cases, the diagram convention and style must come from `sdd-design-doc-mermaid`.

## Recommended document structure

Use the template from `skills/sdd-doc/templates/dm-flow-doc-template.md` as the authoritative structure. The key sections are:

```markdown

## 10. Reutilización del DM

```

## Index and navigation

When a new document is created, also update:

- `docs/flows/README.md`

to add a short link to the newly documented flow.

If `docs/README.md` or `docs/flows/README.md` do not exist yet, create them as part of the same task.

## Final rule

The documentation in `docs/` must be useful to another developer without depending on the original conversation.

If someone opens only the `.md`, they must be able to understand:

- which flow was investigated
- how it starts
- which services the DM calls and in what order
- how response parameters from one service feed the next
- what data moves through it
- which DM performs the real operation
- where the components come from (package vs app-local vs platform)
- how it continues
- where it ends
- which part is the happy path and which part is technical-error handling or logDown
