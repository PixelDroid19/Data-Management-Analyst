# Planning Contract for SDD

## Purpose

This file defines how the AI must **plan** an investigation before executing or closing a trace.

Its role is to avoid two classic problems:

- investigating chaotically
- answering too early with incomplete evidence

## When to use it

Apply it whenever the request involves a flow search, reconstruction, or validation.

Especially when:

- the entry point is still ambiguous
- more than one DM is possible
- navigation spans multiple pages
- intermediate channels or payloads exist
- the happy path must be separated from technical-error handling

Do not skip steps: in this repository, the investigation order matters because the visible UI, the host page, and the real DM may be spread across multiple pages.

## Base planning phases

### 1. Repo profiling

Goal: confirm whether the workspace contains the structure needed for the investigation.

Expected delivery:

- map of available sources of truth
- gaps that may block the trace
- recommended next phase

### 2. Entry discovery

Goal: resolve the initial clue to a candidate view, route, or host page.

Expected delivery:

- shortlist of candidate pages or candidate flows
- first verifiable evidence chain
- explicit note when the result is still `[INFERRED]` or `[NOT FOUND]`

### 3. Scope specification

Goal: turn the request into something demonstrable and unambiguous.

Expected delivery:

- what must be proven
- what stays out of scope
- which open questions remain

### 4. Operational checklist

Goal: transform the scope into small, executable steps.

Expected delivery:

- tasks grouped by phase
- dependencies between tasks
- evidence expected from each task

### 5. Application

Goal: reconstruct the real circuit with code evidence.

Expected delivery:

- traced flow
- real payload
- **service call chain**: which services the DM calls, in what order, what each returns, how response params feed the next request
- channels
- downstream pages
- classification of happy path and errors
- component origin analysis (package vs app-local vs platform)

### 6. Verification

Goal: validate that the investigation is actually closed.

Expected delivery:

- verdict (PASS / PASS WITH GAPS / FAIL)
- documented gaps
- clear criteria for the next step when something is missing

### 7. Documentation and viewer export

Goal: persist the investigation as developer documentation and generate the HTML viewer.

Expected delivery:

- Markdown doc persisted under `docs/flows/<slug>.md` via `sdd-doc`
- HTML viewer generated by running `python skills/sdd-docs-viewer/scripts/deploy_viewer.py --target-repo <path>`
- Verified: `docs/site/index.html`, `app.js`, `styles.css`, `manifest.json` exist

**This phase is the DEFAULT closing behavior.** The user should NEVER need to request it manually. Skip only when the user explicitly opts out.

## Rules for a good task

Every task must be:

- specific
- evidence-oriented
- tied to concrete files, symbols, or searches
- executable in a single pass
- placed in the correct order

Every task must also state:

- **Read first** — the file(s), symbol(s), or evidence source that must be inspected before any claim
- **Search for** — the concrete token, method, channel, route, or payload field expected
- **Claim only if found** — the conclusion allowed when the evidence appears
- **If absent** — whether to mark `[NOT FOUND]`, branch to another task, or stop the trace at that point

## Read-before-claim gate

Planning is incomplete if it lets a later phase conclude without reading the files that prove the conclusion.

For broad or ambiguous prompts, the plan must first include a candidate-flow or candidate-anchor detection step before it assigns a single definitive path.

## Minimum sequence that must not be broken

- Do not study package internals before confirming the host page.
- Do not accept a payload before reviewing setters, getters, helpers, and transforms.
- Do not close a flow before following channels and downstream pages.
- Do not close without documenting the service call chain (order, responses, param mapping).
- Do not convert a candidate or partial anchor into a confirmed one until the host page and evidence chain have been read.
- Do not assume the first DM found is the principal one.
- Do not treat a logDown DM as if it were the principal DM of the happy path.
- Do not limit the answer to theoretical package capabilities when real app usage says otherwise.
- **Stay focused on the flow the user asked about.** Do not drift into tangential flows.

## Done rule

The planning is good only if it leaves `sdd-apply` with a clear path, `sdd-verify` with verifiable criteria, and `sdd-doc` + `sdd-docs-viewer` as the default closing phase.
